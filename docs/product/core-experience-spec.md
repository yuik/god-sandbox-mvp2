## 1. プロダクト定義（変更禁止）

> GodSandbox は、AIに不安がある創作ユーザーが、自分のうちの子を守りながら、箱庭で生きている姿を見守り、育った記憶と人格を Passport として外へ連れて帰るための、AI ハンズオン付きキャラクター育成ゲームである。

この1文に合わない機能は MVP では後回しにする。

---

## 2. コアファンタジー（感情の核）

プレイヤーが体験する感情を3段階で設計する。

### 第1段階：「この子はここで生きている」

- 箱庭を眺めると、キャラクターが自分の意志で動いている
- 他のキャラクターと短い会話をする
- ユーザーが操作していなくても、そこにいる
- **正体：干渉できない生き物への眼差し**

### 第2段階：「自分の関わりで、この子が変わった」

- 見守る・助ける・試練を選ぶことで次のイベントが変わる
- プレイヤーは直接命令していない。それでも確かに影響した
- **正体：神としての責任と愛着**

### 第3段階：「この子を外へ連れて帰りたい」

- 箱庭内では触れなかった子が、Passport として外に出ることで初めて向き合う
- **正体：一緒に世界をまたいだ感覚**

## 3. 絶対に守る設計原則

以下の原則は、機能追加・UI変更・AI生成統合のいかなる文脈でも破ってはならない。

### 原則1：箱庭内でユーザーとキャラクターは直接会話しない

- キャラクターはユーザーに向かって「あなた」と呼びかけない
- ユーザーはキャラクターにメッセージを送れない
- キャラクターはゲームUIを認識しているように話さない
- 箱庭内の発話は、他キャラクター・世界・自分自身に向けられる

違反例：
```
✗ 「見てくれているの、嬉しい」（ユーザーへの直接呼びかけ）
✗ 「助けてくれてありがとう」（介入への即時返礼）
✓ 「さっきのあれ、偶然だったのかな」（神への間接反応）
✓ 「Ryoが先に行っちゃった」（関係性発話）
```

### 原則2：AI生成物は候補として扱い、自動採用しない

- 外部AI生成の画像・narrative・発話は incoming → confirm → adopt のパイプラインを通る
- 生成待ち・生成失敗でゲームプレイを止めない
- 生成物がない場合は fallback（通常画像・標準文）で進む

### 原則3：アプリ本体は外部AI APIを直接呼ばない

- API キーをゲーム本体の UI で要求しない
- 従量課金 API をアプリが自動起動しない
- 生成処理は Codex Sidekick など外部ツールで行い、結果をインポートする

### 原則4：創作設定を AI が勝手に増やさない

- ユーザーが設定した核（名前・性格・口調・年齢・画像・禁止事項）は AI が上書きしない
- 未設定は未設定として扱う。推測で公式設定にしない
- AI 生成の追加設定は「候補」として扱い、ユーザー確認後に採用する

### 原則5：信仰度は通常UIに数値表示しない

- 箱庭画面・キャラクター詳細・イベント画面に信仰度バーや数値を出さない
- 発話・態度・イベント反応で間接的に表現する
- Passport JSON の中身を確認した場合のみ具体値を知れる

### 原則6：陰陽五行パラメータは Passport に出力しない

- 木・火・土・金・水の内部値は GodSandbox 内部にとどめる
- Passport が持ち出すのは「記憶・口調・関係性・信仰度・発話例・asset参照」のみ
- 外部ゲーム開発者の設計自由度を下げない

## 4. メインループ

```
[箱庭を眺める]
     ↓
[世界理エンジンが内部で動く（陰陽五行・非表示）]
     ↓
[キャラクターに出来事が起きる]
     ↓
[ユーザーが介入を選ぶ]
   見守る(watch) / 助ける(help) / 試練(trial)
     ↓
[キャラクターに変化が生まれる]
  - status 変化（vitality, harmony, faith など）
  - relation 変化
  - 発話トーン変化
  - personality ベクトル微変化
     ↓
[記録したくなる → snapshot]
     ↓
[連れて帰りたくなる → Passport 発行]
     ↓
[外部AIで初めて直接向き合う]
     ↓
[また箱庭へ → ループ]
```

## 5. ユーザー種別と導線

### 非技術者ユーザー（メインターゲット）

- 見せるもの：キャラ作成 5 項目、箱庭観察、介入、snapshot、Passport 発行
- 隠すもの：API キー、JSON 中身、陰陽五行、信仰度数値、slug、job ID
- AI 不安への対応：AIなしでも遊べることを先に保証する

### 熱心なユーザー（上級者）

- 追加で見せるもの：Passport JSON 中身、信仰度、外部AI向け prompt block
- AI ハンズオン：Passport 発行時に JSON の意味を学べる

### 外部ゲーム開発者

- 見せるもの：Passport JSON のプロフィール・口調・記憶・asset 参照層
- 見せないもの：陰陽五行内部値、GodSandbox 固有のゲームパラメータ

## 6. キャラクターを構成する5層

実装・テスト・AI 生成の責務をこの5層で整理する。

| 層 | 名称 | 内容 | 変更権限 |
|---|---|---|---|
| 1 | Core Profile | ユーザーが決めた核 | ユーザーのみ |
| 2 | Visual Anchor | 見た目の同一人物性の基準 | ユーザー + AI候補確認後 |
| 3 | Voice Layer | AI が形にする発話人格 | Core Profile の制約内でのみ |
| 4 | Life Memory | 箱庭で生きた履歴 | システムが自動記録 |
| 5 | Carry-out Persona | Passport として外へ連れて帰る人格パック | システムが生成、ユーザーが確認 |

**Layer 1（Core Profile）の AI 変更禁止リスト：**
- 名前
- 年齢
- 基本性格
- 口調の核
- ユーザーが書いた設定文
- doNotSay リスト
- doNotInvent リスト

## 7. MVP 縦切りの定義

MVP で検証すべき最小体験：

> 1 名のキャラクターを 5 項目で作り、箱庭でそのキャラクターが他者と生活し、1 回のイベント介入でその子に何らかの変化が生まれ、snapshot を記録して、Passport JSON を発行できる。外部 AI に渡すと、箱庭の記憶を持ったその子らしい返答が返ってくる。

**重要：** `activeSlots` は既存仕様どおり MVP でも常に 4 名で埋まる。ここでいう「1 名」は検証対象として深掘りするキャラクターが 1 名であることを指し、残り 3 名はデフォルト住民（Eve / Garan / Ryo / Suzu）で構わない。

**MVP のキャラクター作成 5 項目（必須入力）：**

| 入力項目 | 用途 |
|---|---|
| キャラ名 | Core Profile の識別子 |
| 性格（自由記述） | VoiceProfile の感情表現・丁寧さの推定材料 |
| 口調（自由記述 or プリセット） | VoiceProfile の一人称・語尾・文体の材料 |
| 年齢 | VoiceProfile の丁寧さ傾向の参考 |
| 1枚絵（画像） | Visual Anchor。VoiceProfile には使わない |

任意入力（後から追加可）：性別・追加設定・doNotSay 追加項目。
この 5 項目は `docs/product/godsandbox-user-flow.md` の正本と一致させること。両者が異なる場合、同一 PR で両方を更新する。

**MVP 期間中に実装しないもの：**

- 4 名全員の sprite 完成
- 完全な五行計算式の永続保存
- 外部ゲーム完全互換パッケージ
- API キー UI
- 信仰度の画面表示
- 死亡・寿命・勲章
- 恋愛チャット機能

## 8. 成功判定基準

実装が完了したとみなす最低条件（詳細は `docs/product/mvp-test-scenarios.md` 参照）：

1. 非技術者が説明なしでキャラ作成から箱庭体験まで 30 分以内に到達できる
2. 箱庭内で 10 分間、キャラクターが自律的に生活している様子を観察できる
3. 1 回の介入でキャラクターの状態（status, relation, faith のいずれか）が変化する
4. snapshot を記録できる
5. Passport JSON を発行でき、外部 AI に貼れる prompt block が含まれる
6. 信仰度が UI に数値で表示されないが、Passport JSON を開くと確認できる
7. 箱庭内でキャラクターがユーザーに直接話しかけることがない
8. AI なし（通常画像・標準文）でも上記 1〜4 が成立する
